## 2. プロジェクト構造と役割：実行境界の厳格な分離
「どこで動いているコードか」をファイル名で物理的に判別し、スパゲッティ化を防止します。

### ■ アプリケーション・コア (`src/`)
| フォルダ / 拡張子 | 役割 (責務) | 実行環境 | 不具合時の調査対象 |
| :--- | :--- | :--- | :--- |
| `*.server.ts` | **機密・DB操作**。D1, Bindings, 認証。 | サーバー専用 | **データ欠如、SQLエラー、500エラー** |
| `*.client.ts` | **動的・Fetch**。イベント, UI更新。 | ブラウザ専用 | **ボタン無反応、画面更新不良** |
| `*.ui.tsx` | **純粋な見た目**。ロジック抜きのJSX。 | 共通(SSR) | **レイアウト崩れ、表示ミス** |
| `services/` | データアクセス層。 | Server | DB疎通・ロジック |
| `hooks/` | フロントエンド論理。 | Client | 状態管理・非同期通信 |
| `types/` | データの法律。型定義(Zod)。 | 共通 | 型不一致・バリデーション失敗 |

---

## 3. 開発フロー：三段階のステップ
1. **[思考] Sandbox**: `src/_sandbox/` でロジックの断片を試作。
2. **[検証] Proto (本リポジトリ)**: 正規フォルダに組み込み、システム整合性を確認。
3. **[昇格] Production ( )**: 検証済みのコードを本番用リポジトリへ移植。

---

## 4. デバッグと疎通の掟 (The Rule of Traceability)
不具合の所在を「一瞬で特定」するために以下のルールを遵守すること。

### ① 最小経路の優先 (Atomic First)
複雑な部品化（Hook化）の前に、必ず「単一ファイル」または「API単体」でDBとの疎通を確認する。画面が動かない時は、まずブラウザで直接 API URL（`/api/xxx`）を叩き、生データを確認すること。

### ② ログの指紋 (Log Signatures)
出力場所に以下の接頭辞を付け、調査すべきログ（ターミナル vs ブラウザコンソール）を明確にする。
- `console.log("[Server:API] ...")` -> サーバー・DB側の問題
- `console.log("[Client:Hook] ...")` -> ブラウザ・JS実行側の問題

### ③ 実行境界の死守
`D1Database` 等のサーバー専用オブジェクトをブラウザ側コード (`*.client.ts`) に持ち込まない。ブラウザは常に API エンドポイントを経由してデータを受け取る「疎結合」を維持する。

## 5. セキュリティ・ポリシー
- **秘匿情報の隔離**: APIキー等は `wrangler secret put` で登録すること。
- **型と検証の二重化**: 静的な型(Types)で開発ミスを防ぎ、動的な検証(Schemas)で不正データからシステムを守る。
