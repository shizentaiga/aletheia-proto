# 03_Database_Design: データベース設計（v1.1）

## 1. データの役割と設計思想
ALETHEIAのデータベースは、単なる記録場所ではなく、**「外部API依存からの自立」と「正確な地点情報の昇華」を支えるインフラ**として設計されています。
Cloudflare D1（SQLite）の特性を活かし、人間が直感的に理解・修復できるシンプルな構造を維持します。

* **インフラ**: Cloudflare D1 (SQLiteベース)
* **設計原則 (Zen)**: 
    - **Single Source of Truth**: `booked_by` の有無が予約の唯一の真実。
    - **非依存性**: 外部Place IDをフックにしつつ、主キーには自社発行の `service_id` を使用。
* **ID体系**: 時系列ソートと分散環境での衝突回避を両立するため、原則として **`ULID`** を採用。

---

## 2. 主要エンティティの定義

### ① ユーザー (users)
権限管理（Role）と、データの「昇華」を担う有料プラン（Plan）の紐付けを管理します。
* **実装仕様**:
    - `user_id`: 内部管理用ID (**ULID**)
    - `google_id`: OAuth連携用。NULL許容により、後からの紐付けに対応。
    - `role`: `admin` (管理者), `owner` (店舗主), `user` (一般)
    - `plan_id`: 権限レベルを決定するプラン（`free` / `editor_pro`）

### ② 地点・サービス (services)
「初期シード（外部データ）」から「自社資産」へと進化する中心的なテーブルです。
* **ビジネス上の意味**:
    - `ext_place_id`: 重複登録防止のための外部識別子（Google Place ID等）。
    - `verification_level`: 0 (未検証), 1 (座標確定), 2 (オーナー確認済)。
    - `geohash_9`: 前方一致検索による超高速な近傍探索を実現。
* **主要カラム**:
    - `service_id`: 自社発行ID (**ULID**)
    - `owner_id`: NULL許容。店舗オーナーが名乗り出た際に紐付け。
    - `title` / `address`: ユーザー修正提案により随時上書きされる。

### ③ 時間枠・在庫 (slots)
サービス提供者の「時間」を管理します。
* **予約ロジック**: 
    - `booked_by IS NULL` を空き枠の条件とし、`version` カラムによる楽観的ロックで同時予約の衝突を物理的に排除する。
* **主要カラム**:
    - `slot_id`: **ULID**
    - `start_at_unix`: 開始時刻 (UNIXタイムスタンプ)
    - `booked_by`: 予約した `user_id`

### ④ データの昇華機構 (proposals & supports)
情報の質を有料ユーザーの「熱量」によって高める仕組みです。
* **proposals**: 店舗情報の修正案（住所の微修正、Wi-Fiの有無等）を記録。
* **proposal_supports**: 提案に対する他の有料ユーザーからの支持（賛成）を記録。
* **ガバナンス**: 一定数の支持が得られた提案をシステムが自動反映（または管理者が一括承認）する。

## 3. 安全性と整合性プロトコル

* **削除・退会ポリシー**:
    - ユーザー削除時、提案支持等は連鎖削除（`CASCADE`）される。
    - ただし「未完了の予約（Slots）」が存在する場合、物理削除を `RESTRICT` でブロックし、サービス上の事故を防ぐ。
* **パフォーマンス最適化**:
    - `geohash_9` へのインデックスにより、`LIKE 'abcde%'` 形式のクエリで範囲検索を最適化。
    - 部分インデックスを活用し、論理削除（`deleted_at`）されていない有効データのみを高速走査する。

---

## 4. テクニカル・ガバナンス（実走ルール）

1. **Schema-Sync**: 
    - `schema.sql` (DB層) と `types/*.ts` (TypeScript/Zod層) の定義を常に同期し、アプリケーション層でのランタイムエラーを防止する。
2. **原子性（Atomicity）**: 
    - 予約処理や決済（Stripe連携予定）は、D1のトランザクションを用いて「全か無か」の実行を保証する。
3. **データ・クリーンアップ**: 
    - `apply_seeds.sh` によるシード投入時、`_core.sql`（基盤）から順に投入し、外部キー制約エラーを構造的に回避する。
