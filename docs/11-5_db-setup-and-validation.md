# D1ローカルデータベース構築および疎通確認（Technical Reference）

本資料では、プロジェクト『 』におけるエッジデータベース（Cloudflare D1）のローカル環境構築、およびデータアクセス層の検証手順を記録する。

---

## 1. データベース設計思想（Physical Schema Design）
スケーラビリティと将来のデータ匿名化（Privacy by Design）を両立させるため、以下の設計方針を採用した。

* **非正規化への許容**: 
    - **技術的意図**: 初期フェーズでは `users` テーブルにセッション情報（refresh_token）を包含し、処理負荷（JOIN）を最小化。
    - **初心者向け解説**: 情報をあちこちに分けず「1枚の名簿」にまとめることで、素早く読み書きできる設定にしています。
* **緩やかな制約（Flexible Constraints）**: 
    - **技術的意図**: `email` の `NOT NULL` および `UNIQUE` 制約を排除。
    - **ビジネス上の意図**: 1人のオーナーによる複数アカウント運用を許容し、将来的な個人情報（PII）破棄プロセスにも柔軟に対応可能とする。
    - **初心者向け解説**: 「メールアドレスが必須」というルールをあえて外すことで、ユーザーが情報を消したい時や、複数の店舗を使い分けたい時に柔軟に対応できるようにしています。

---

## 2. 実装プロセス（Implementation Workflow）

### ステップ①：DDLによるスキーマ定義（schema.sql）
SQLite 互換の指示書（DDL）を記述。初期データとしてモックレコードを挿入（INSERT）し、動作確認を容易にする。

* **Created_at**: `DEFAULT CURRENT_TIMESTAMP` により、登録日時を自動記録。
* **Audit Logs**: `last_login_at` を備え、セッションの自動延命（ローリング・セッション）の判定基準とする。
* **初心者向け解説**: 誰がいつ登録したか、最後にいつ来たかを自動でメモする仕組みを組み込みました。

### ステップ②：Wrangler によるローカル・プロビジョニング
Cloudflare の開発ツール `wrangler` を使用し、自分のPC内（ローカル環境）にデータベースの実体を生成する。

```bash
# ローカルDBへの設計図適用とテストデータの投入
npx wrangler d1 execute  -db --local --file=./schema.sql
```

* **初心者向け解説**: この一行の命令で、PCの中に「情報の置き場所」が完成します。

### ステップ③：データアクセス層（DAL）の検証（sandbox/test02）
プログラム（Hono）を通じてデータベースを操作し、正しくデータが流れるかを確認するフェーズです。

* **全件取得**: `SELECT * FROM users` による名簿全体の呼び出し。
* **プレースホルダ利用**: セキュリティ（SQLインジェクション）対策を施した特定ID検索。
* **初心者向け解説**: 「全員分見せて」「この人だけ探して」という命令が、安全に正しく動くかをテストしました。

## 3. 疎通確認の結果（Validation Results）
以下のテスト専用URL（サンドボックス）を通じて、正常動作を確認済みです。

* **全体表示テスト**: `GET /sandbox/test02`
    * データベース内の全レコードがJSON形式で応答されることを確認。
* **個別検索テスト**: `GET /sandbox/test02/:id`
    * 特定のIDを指定した際、該当する1名のみが瞬時に抽出されることを確認。

## 4. エンジニア向け注記（Notes for Future Scaling）
今後の拡張と運用に向けた技術的メモです。

* **Local State**: セッション検証等でDBを初期化する場合、上記コマンドを再実行することで瞬時に Seed データ（テスト用初期値）ありの状態へ復元可能。
* **Migration Strategy**: 登録者が 100 名を超えた段階で、`--remote` フラグを用いた本番環境への移行、および Google OAuth の制限解除申請（Production移行）を検討する。
