# 技術資料：CSRF (Cross-Site Request Forgery) 対策指針

## 1. CSRF (Cross-Site Request Forgery) とは
CSRF（シーサーフ）とは、ユーザーがログイン中のWebサイト（本サイト）に対し、悪意のある第三者のサイト（攻撃サイト）を経由して、ユーザーが意図しないリクエスト（書き込み、購入、退会など）を強制的に送信させる攻撃手法です。

ブラウザが「特定のドメインに対して発行されたクッキー」を、そのドメインへのリクエストに自動的に付与して送信するという特性を悪用します。

## 2. 対策しないことによる影響
ログイン済みの有効なセッションがある状態で攻撃を受けると、以下のような事態を招く恐れがあります。

- **意図しない退会処理**: 本人が知らないうちにアカウントが削除される。
- **設定情報の改ざん**: メールアドレスやパスワードを変更され、アカウントを乗っ取られる。
- **なりすまし投稿/購入**: ユーザーの名義で不適切なコンテンツが投稿されたり、商品が注文されたりする。

## 3. 今回の暫定対策：SameSite属性の導入
今回のプロジェクトでは、クッキーに「Lax属性」を設定しました。これにより、ブラウザが「外部サイトから送られてくる怪しいリクエスト」にはクッキーを添付しないよう制限してくれるため、基本的なセキュリティ（CSRF対策）が強化されます。

### 修正内容
```
(setCookie 設定箇所)
setCookie(c, AUTH_CONFIG.SESSION_COOKIE, user.user_id, {
  path: '/', 
  httpOnly: true, 
  secure: true, 
  maxAge: 3600, 
  sameSite: 'Lax' // ← CSRF対策として追加
})
```

### Lax属性の効果
- **クロスサイトでの制限**: 他のドメイン（攻撃用サイトなど）から送信される POST リクエストや、画像埋め込み等による GET リクエストにはクッキーが付帯しなくなります。
- **利便性の維持**: 外部サイトのリンクをクリックして自サイトへ遷移する（トップレベルの遷移）場合のみクッキーが付帯するため、ユーザーはログイン状態を維持したままアクセス可能です。

## 4. 本格的に対策する場合の方法
システム規模や要件が拡大した場合には、以下の「二重の防御」を検討します。

### A. CSRFトークン（Anti-CSRF Token）の利用
1. サーバーがフォーム表示時に、予測困難な「使い捨てトークン」を発行し、セッションに保存。
2. クライアントは POST 送信時にそのトークンを hidden パラメータや HTTP ヘッダーに含める。
3. サーバー側で、送信されたトークンと保持しているトークンが一致するか検証する。

### B. Custom Header (Same-Origin) 検証
JavaScript（fetch/axios）による非同期通信の場合、特定のカスタムヘッダー（例: `X-Requested-With`）を必須とし、サーバー側でその存在をチェックします。これはブラウザの CORS ポリシーにより、別ドメインからのリクエストでは（許可なしに）カスタムヘッダーを付与できない性質を利用したものです。

## 5. その他・留意点
- **GETリクエストの冪等性（Idempotency）の保持**: データの更新や削除は必ず POST/DELETE 等で行い、GET リクエストで状態を変化させないことが基本です。
- **Secure属性**: `sameSite: 'None'` を使用する場合は `Secure` 属性が必須となりますが、セキュリティの観点から可能な限り `Lax` または `Strict` を優先します。
