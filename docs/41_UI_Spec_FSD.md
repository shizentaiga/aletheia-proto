# エリア選択 UI/UX 仕様書 (Full-screen Stack Drilldown) Ver 0.5

## 1. コンセプト
「深さは横、モードは縦」の原則に基づき、モバイルブラウザにおいてネイティブアプリと同等の操作感を実現する。特に「履歴管理の整合性」と「データ通信の効率化」に重点を置く。

## 2. 階層設計とデータ構造

### 2.1 最大階層数の定義 (MAX_DEPTH)
UXの複雑化とクエリ負荷を避けるため、階層は最大4階層に固定する。
- Level 1 (地方): 北海道, 関東, 関西...
- Level 2 (都道府県): 東京都, 神奈川県...
- Level 3 (市区町村): 新宿区, 渋谷区...
- Level 4 (エリア/町域): 西新宿, 道玄坂...（※将来拡張用）

### 2.2 API レスポンス定義 (/api/area-stats)
JSON形式（フロントエンドとバックエンドの型定義を一致させる）
[
  {
    "id": "13",
    "name": "東京都",
    "count": 2301,
    "hasChildren": true
  }
]

## 3. インタラクション & 履歴管理（履歴の一本化）

### 3.1 履歴スタックの保護と上書き運用
ドリルダウン内部の遷移がブラウザ履歴を汚染するのを防ぎ、かつSPAの整合性を保つため、以下の運用を徹底する。
- **オーバーレイ展開時**: `history.pushState({ type: 'drilldown_open' }, ...)` を1回だけ実行。
- **内部の階層遷移**: 履歴スタックを増やさず `history.replaceState({ type: 'drilldown_nav', depth: N }, ...)` で現在の状態のみを更新。
- **検索確定 / 閉じる操作**: 
    - `history.back()` は使用せず、最終的な検索状態（または初期状態）を `history.replaceState({ type: 'search', area: selectedArea }, ...)` で上書きする。
    - その後、スクリプト制御でオーバーレイを閉じる。
👉 **効果**: ブラウザの `popstate` 発火による意図しない画面遷移やHTMXの再リクエストを回避しつつ、ドリルダウンを「1つの履歴上の出来事」として完結させる。

### 3.2 アニメーションの最適化
- Duration: 200ms (180ms〜220msの範囲で調整)
- Easing: cubic-bezier(0.33, 1, 0.68, 1)
- 状態維持: オーバーレイを閉じて再度開いた際は、「前回の到達階層」を維持して表示。

### 3.3 エッジスワイプの競合回避
iOS Safariのシステムジェスチャーとの競合を避ける判定。
- `touchstart.x < 16px` かつ `scrollLeft === 0` かつ `stack.length > 1` の場合のみ発動。

## 4. データ・クエリ仕様

### 4.1 「全域」選択の定義
[階層名]（全域）を選択した場合、その階層の子孫要素すべてを含む結果を返す。
- SQLイメージ: `WHERE area_code LIKE '13%'`（前方一致による集計）

### 4.2 キャッシュ戦略
D1の負荷軽減のため、area-stats APIは以下のいずれかでキャッシュする。
- Cache API / KV: 5分〜30分保持。
- 集計テーブル: `area_stats_cache` テーブルを用意し、データ更新時に更新。

## 5. 状態保持とスクロール復元

### 5.1 スクロールキーの厳密化
階層IDだけでなく、検索条件のハッシュを組み合わせたキーで管理する。
- Key: `layer_id + query_params_hash`
これにより、検索内容が変わった際に誤った位置に復元されるのを防ぐ。

## 6. 技術スタックの責務分離（SoC）

| 技術 | 主な責務 |
| :--- | :--- |
| Hono | ルーティング、HTML Fragment（サーバーサイド描画）、JSON API提供 |
| HTMX | 検索結果エリアの非同期差し替え、HX-Trigger によるJSとの連携 |
| Vanilla JS | スタック管理、アニメーション、スワイプ、History API、スクロール保持 |
