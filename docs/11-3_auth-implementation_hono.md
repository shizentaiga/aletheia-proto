# 11-3_Google_OAuth2_with_Hono

このドキュメントでは、Cloudflare Workers (Wrangler) 上で Hono を使い、Google 認証を実装する手順を解説します。

## 1. Wrangler とは何か？
**Wrangler** は、Cloudflare Workers を開発・テスト・デプロイするためのコマンドラインツール（CLI）です。
このプロジェクトでは「ローカルサーバー」の役割を果たします。`.dev.vars` から環境変数を読み込んだり、自分のPC上で Cloudflare の環境をシミュレートしたりしてくれます。

---

## 2. 前提条件
開始する前に、[Google Cloud Console](https://console.cloud.google.com/) で以下を取得している必要があります。
- **クライアント ID** と **クライアント シークレット** (OAuth 2.0 クライアント)
- **承認済みのリダイレクト URI**: 
  ローカルテスト用に `http://localhost:8787/test/google` を設定してください。

---

## 3. 実装時にハマりやすいポイント（重要！）

### ⚠️ 注意：`.dev.vars` の置き場所
これが最も多いミスです。秘密鍵を保存する `.dev.vars` ファイルは、必ずプロジェクトの **「ルートディレクトリ（一番上の階層）」** に置いてください。
- ✅ **正しい:** `your-project/.dev.vars` (`package.json` と同じ並び)
- ❌ **間違い:** `your-project/src/.dev.vars`
*もし `src/` の中に入れてしまうと、Wrangler がファイルを認識できず、環境変数が "MISSING"（未検出）になります。*

### ⚠️ ログイン用と受け取り用のパスを分ける
「ログインを開始するURL」と「結果を受け取るURL」を全く同じにするとエラーになります。
- **ログイン開始:** `/test/login` (ユーザーがここをクリックして Google へ飛ぶ)
- **結果受け取り:** `/test/google` (Google がユーザーを戻してくる場所)

### ⚠️ セキュリティ：.gitignore への登録
`.dev.vars` には本物のクライアントシークレットが含まれています。これを GitHub 等に公開しないよう、必ずプロジェクト直地の `.gitignore` ファイルに以下の 1 行を追加してください。
- ✅ **追記内容:** `.dev.vars`
*これを忘れると、世界中の誰でもあなたの Google Cloud プロジェクトを操作できてしまいます。*


---

## 4. デバッグ方法（動かない時の調べ方）
もしエラーが出たら、ターミナル（コンソール）のログを確認しましょう。

1. **環境変数のチェック:**
   `console.log` を使って、`c.env.GOOGLE_CLIENT_ID` が `OK` か `MISSING` かを表示させます。
2. **クエリパラメータのチェック:**
   Google から戻ってきた際、URLの末尾に `?code=...` が付いているか確認します。
3. **ミドルウェアの実行:**
   受け取り側のルート (`/test/google`) で、`googleAuth` ミドルウェアが「実行」されているか確認してください。単に定義するだけでなく、実行しないとユーザーデータは解析されません。

---

## 5. 最終的な確認結果
成功すると、ターミナルとブラウザに以下のような結果が表示されます。

### ターミナルログ（成功パターン）
```text
CLIENT_ID: OK
[wrangler:inf] GET /test/login 302 Found
--- Callback Reached ---
Query Params: { code: '4/0Aci...', ... }
User Data Status: FOUND
[wrangler:inf] GET /test/google 200 OK
```

## 6. 次のステップ：セッションの永続化
現在の実装は「認証してメールアドレスを表示して終了」という単発の処理です。実際のサービス（ ）として運用するには、以下の拡張が必要になります。

1. **D1 データベースへの保存**: 取得した Google ID やメールアドレスを D1 の `users` テーブルへ保存し、リピーターを識別する。
2. **Cookie / JWT によるログイン維持**: ブラウザを閉じても「ログイン中」の状態が続くよう、認証情報をクッキーにセットする処理を追加する。
